### How the pieces fit together

1. **Startup**: `main.py` launches Uvicorn with `app` from `app.py`.  `app` registers routers from `api/` and sets up dependencies and middleware.
2. **Configuration**: `database/config.Config` reads env vars; validation happens on startup.  `dependencies.py` uses `Config` for feature flags.
3. **Request handling**: Incoming HTTP requests hit `api` routers; dependencies yield database sessions, current user, etc.  Schemas enforce validation.
4. **Business logic**: API code delegates to `services/` which manipulate ORM models or call `domains/` algorithms.  Caching, notifications, payments, and external API calls are all abstracted here.
5. **Persistence**: SQLAlchemy sessions talk to the Postgres database defined by `Config`.  Redis is used for caching and session blacklisting.
6. **Background & utilities**: Scripts and ETL code are used outside of request flow to manage data, perform migrations, or run smoke tests.

### Usage guidance

- To add a new API endpoint, create a router module under `api/`, define Pydantic schemas in `schemas.py` (or a new file), and add business logic to an appropriate service.  Use `dependencies.py` to inject required services.
- For new domain algorithms, add a package under `domains/` and call it from `services` or `dependencies` as needed.
- Keep the `database/` package focused on models and configuration; migration scripts live in `alembic/`.
- Development helpers belong in `scripts/` or top‑level, not under packages imported by the application.
